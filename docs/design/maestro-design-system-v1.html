排产调度 Agent
的视觉系统重构

一套面向制造车间的 Agent 界面。它要同时承载两种阅读方式——对话是被读的，工单与甘特图是被扫的。色彩预算全部花在「这条产线现在出了什么事」上：一个蓝、一个绿、一个琥珀、一个红，各自只做一件事。

01Brand

商标：指挥棒

Maestro 是指挥家。图标画的就是指挥这个动作——握棒的支点、扬起的指挥棒、两道逐渐淡出的手势轨迹。不是抽象几何，也不是又一个字母 M；一眼能读出"有人在打拍子"。两道弧同时暗示三个引擎被收在同一拍上。

主标记 · 尺寸与场景
Maestro排产调度 Maestro Maestro

字标不再挤。原先 Geist 600 配 −0.02em 负字距，在 14px 下字母几乎粘连。现在字标单独用 −0.005em（比正文标题松得多），字号提到 17px，与图标间距 10px；徽章圆角 9px 与标记内部曲率对齐。副标「排产调度」用 +0.18em 字距的 9.5px 小字压在下方，把主字标的重量托起来。

图标族 · 全部为本产品重绘
附件 技能 发送 直接执行 需确认

技能重画成六边形芯片加三根引脚——技能是可插拔的能力包，不是魔法棒。发送重画成纸飞机：机身实心，折痕用底色描出，比向上箭头更明确地说"这条消息会飞出去"，也不再和工具栏里其他箭头类图标混淆。

中文字重与字距规则

✕ 不要这样
装配二段产能负载 96%

PingFang SC 没有真正的 SemiBold，font-weight:600 会触发合成伪粗体，笔画糊成一团；−.02em 的负字距是按拉丁字母调的，套到汉字上会挤压字面。

✓ 这样
装配二段产能负载 96%

中文字重封顶 500（PingFang Medium 是真字重），字距归零。拉丁与数字仍走 Geist 600 / −.02em，两者在同一行里靠字重而非字距区分。

输入框 · 附件最左 · 模式 · 发送最右
描述排产 / 调度 / 查询需求，或输入 / 使用技能…

附件是"往对话里加东西"，属于输入源，所以钉在最左，用一道分隔线与技能/路由这类"改变本次发送行为"的芯片隔开。模式选择器在发送键左边——它是发送前你要确认的最后一件事。发送是终点，钉在最右，实心蓝，全界面唯一一个蓝色方形图标按钮。

模式选择 · 默认 vs 完全访问
Agent 先给出方案与待执行动作，任何写操作都停在 ActionGate，等你逐条确认。琥珀色的「确认执行」按钮会出现在对话里。
CONFIRM · dispatch_work_order
Agent 可直接写入 MES，不再停下等确认。琥珀色那一步整个消失——你在对话里看到的是「已执行」，不是「请确认」。
已执行 · dispatch_work_order

为什么「完全访问」的芯片是琥珀不是灰。它绕过 ActionGate 的 requires_confirmation 门控，是这个界面里权限最大的一次点击。默认模式对应后端的 plan，完全访问对应 auto——ComposerMode 类型已存在，本次只换外观与文案。切到完全访问时弹一次确认，之后仅对当前会话生效，不写入全局设置。

05Dialog

弹框：平台规范设置

这是配置整个 Agent 平台规则的地方，所以它不是随手的确认框，而是一个有分区、有解释、有后果提示的正式对话框。最大宽 560px，标题左对齐，主操作在右下——不居中，不加大图标，不用巨型标题。

平台规范设置

这些规则对所有会话生效，并决定哪些动作可以由 Agent 自动执行。

关闭「工单下发需人工确认」后，B 线的 2 条待确认任务令将在下一次巡检时自动执行。

变更即时生效，写入 ~/.maestro/settings.json

为什么保存键是蓝不是绿：保存设置属于导航/沟通类动作，它本身不写 MES。绿色只留给真正改变车间状态的执行。这张弹框里唯一可能造成后果的地方，是那张琥珀色影响提示卡——它出现在保存键之前，不是之后。

无障碍：role="dialog" + aria-modal + aria-labelledby/describedby；打开时焦点移入、关闭时归还给触发元素；Esc 关闭；开关用 role="switch" 与 aria-checked，不是裸 div。

08Migration

落地影响与技术栈判断

结论：复用现有 frontend/ React + Vite 项目，只换 token 层与组件层。依据是这个应用的核心价值全在逻辑里——SSE 流式对话、TanStack Query 的会话与知识库 CRUD、ActionGate 的两级授权确认流、澄清路由回环。独立 HTML 演示会把这些全部丢掉，直接违反「原有核心交互逻辑不丢失」这条验收标准。
