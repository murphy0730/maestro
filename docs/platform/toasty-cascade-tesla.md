# 设计文档改进建议 — `docs/platform/scheduling-platform-design.md`

> 目标：审查 v0.1 设计文档，找出**设计层面**可改进的点（架构思路、描述完整性、缺失考虑、可落地性）。
> 方法：通读设计文档 + 对照实际代码（约 4282 行）落地情况，代码问题作为"设计描述不够刚性/不完整→落地打折"的佐证。
> 结论：架构骨架设计扎实（一个入口/两个引擎/事件层/共享底座分层清晰，策略插件化真正开闭），改进点集中在 **路由反模式防护描述不够刚性、非功能性设计系统性缺失、若干架构职责边界模糊** 三类。共 22 条，按优先级分组。

---

## P0 — 设计正确性缺口（已影响核心验收）

### 1. 路由第 0 层「显式命令」设计不完整
- **问题**：设计 5.2 第 0 层只说"斜杠命令 / UI 动作直接路由，route_method=explicit"，但未定义：命令语法（`/排产 <参数>`）、命令与自然语言参数如何结合、explicit 路由的 `RouteDecision` 结构、与第 1 层嵌入路由的优先级协议。
- **佐证**：代码 `router.py` 直接从嵌入层开始，第 0 层完全未实现，`route_method` 的 Literal 也无 `explicit`。描述不刚性→落地时被静默跳过。
- **改进**：补一节"显式命令协议"，明确命令清单、参数抽取规则、explicit 命中后是否仍记录实体、与澄清后路由(选项式)的 route_method 区分。

### 2. query 类定义过宽且锚点描述不够刚性
- **问题**：设计把"概念解释 / 历史查询 / 使用咨询 / 闲聊"全归 query 类，但只在文字里提了一句"必须含概念例句"，未在"路由器核心逻辑"里拆 query 子类，也没说明 query handler 如何内部分流（解释概念 vs 查历史数据 vs 闲聊，处理路径完全不同）。设计 5.2 还专门用示例 E 强调防"排产是什么"误路由，却没把概念锚点写成强约束。
- **佐证**：`routing_examples.yaml` 的 query 类全是数据查询语句（查库存/查订单），无任何概念解释例句；`CLASSIFY_SYSTEM` 对照表也无 query 行。**设计最核心的反模式防护在嵌入层和 LLM 层双重缺锚点**，验收示例 E 可靠性存疑。
- **改进**：把 query 拆为 `query_concept` / `query_history` / `query_help` / `chitchat` 子类（或明确 handler 内部分流规则）；把"概念/闲聊锚点例句"从建议升级为**必须项**并在第 1、2 层都强制；补 query 的易混淆对照例句进第 2 层 prompt。

### 3. margin 阈值 A/B 缺推荐初值与调优方法论
- **问题**：设计 5.2 只说"阈值 A、B 可配置；初期可调高，随话术积累再调低"，未给具体初值、调优数据来源、precision/recall 度量、A 与 B 如何权衡。
- **佐证**：代码实际用 A=0.72、B=0.05（margin 阈值偏低，0.05 几乎不构成"明显甩开"），但文档无依据可循，纯靠实现者拍脑袋。
- **改进**：补推荐初值（如 A=0.75、B=0.10）+ 调优方法论（误判样本集 → 测各阈值下的 precision/recall → 选 operating point）+ 冷启动到稳态的演进路径。

### 4. 决策日志回流机制只提概念未细化
- **问题**：设计 5.2 末尾"决策日志回流补充嵌入示例话术"是有价值的自学习闭环，但未定义：哪些日志算误判（需用户纠正信号）、自动回流还是人工审核、如何防污染（错误例句混入会拉偏向量）、回流频率、回流量上限。
- **改进**：补"回流流程"小节，定义误判识别（用户澄清/纠正即信号）→ 待审池 → 审核 → 进入对应类别示例库（设上限防单一类膨胀）→ 重算原型向量。

## P1 — 架构职责边界模糊（影响可维护性/可扩展性）

### 5. 调度引擎内部子意图分发与入口路由理念不一致
- **问题**：入口路由器精心用语义嵌入反关键词，但设计未约束引擎**内部**的子意图识别。`SchedulingEngine.handle_chat` 用硬关键词（"下发/异常/催/齐套"）细分催料/齐套/下发/异常，无 LLM 兜底，换说法（"赶紧要料"）可能落空。
- **改进**：设计一个统一的"子意图识别"方法（显式命令 + LLM 细分 + 澄清）贯穿入口到引擎内部，文档明确引擎内部分发也应遵循"反关键词猜意图"理念。或允许引擎内部用关键词但需 LLM 兜底 + 记录 route_method。

### 6. validator 与策略内约束的职责重叠
- **问题**：设计 6.5 说"通用硬约束校验（策略特有约束在策略内）"，但产能/产品匹配/不重叠这类约束，策略 `solve` 内部也要建模，存在重复。边界模糊导致约束变更需两处改。
- **改进**：明确两层校验的**目的分层**——`validator` 做"求解后结果兜底校验"（不信任求解器、防 bug），策略 `validate_input` 做"求解前前置可行性校验"（快速失败、省求解时间）；文档列出哪些约束属"前置"哪些属"后置"，避免同一约束两处各写一套。

### 7. 事件层职责过载
- **问题**：设计 8.1 的 `scheduler` 既 `poll_events()` 拉外部事件，又做"预测性齐套扫描"（扫描待开工任务令缺料主动产生事件）。后者本质依赖调度引擎的齐套能力，把"外部事件拉取"和"内部预测事件生成"混在一个组件，职责耦合。
- **佐证**：`handlers.py`/`scheduler.py` 直接 import 调度引擎具体类，事件层→引擎层硬耦合。
- **改进**：拆为「外部事件拉取器」（只 poll/subscribe 外部系统）+「内部预测事件源」（齐套预测，作为独立组件或引擎自身产事件）；事件层只负责总线与分发，不直接 import 引擎具体类（经接口/注册）。

### 8. 排产策略选择层"低置信澄清"的交互设计欠妥
- **问题**：设计 6.4 第 3 层向用户问"这批单属于哪种排产场景？"并带候选策略选项。但普通计划员未必懂"作业车间/流水车间/换型最少"等算法概念，用算法名做澄清选项会造成体验断裂。
- **改进**：澄清选项用**人话描述场景特征**（如"换型频繁、模具受限的注塑" vs "流水线大批量"）而非策略名；或优先按工艺特征/产品线自动推断，仅极度模糊时才问用户，且问的是"产品线/工艺"而非"算法"。

### 9. 事件驱动结果对离线用户的通知投递未设计
- **问题**：事件唤醒调度引擎后产生催料/异常，对**当前无活跃会话**的用户如何送达？设计 7.3/8.1 只说"打印 + 写日志/待办"。这削弱了"事件驱动是调度一等公民"的价值（人不在场就收不到）。
- **改进**：补"通知投递层"接口预留（站内信/邮件/IM/webhook），设计 `NotificationChannel` 抽象，事件处理结果按规则投递给责任人，v0.1 可只实现日志 channel。

## P1 — 非功能性设计系统性缺失

### 10. 缺独立的「安全设计」章节
- **问题**：安全只在 13 节一句话"不要硬编码 API key"。无密钥管理（存储/轮换/最小权限）、无输入校验/注入防护（LLM prompt 注入、用户输入到求解器）、无审计防篡改、无多租户隔离预留。
- **佐证**：`config.py` 硬编码真实密钥作默认值（进源码/版本库），直接违反 13 节——根源是安全只一句话带过、无刚性约束。
- **改进**：新增"安全设计"章节：密钥仅从环境/密钥管理服务注入且默认空；LLM 输出强校验（已部分做到）；审计日志追加只写防篡改；权限 RBAC 预留。

### 11. 可观测性只提结构化日志，缺 metrics 与 trace
- **问题**：设计 2.1 原则 6"全部产生结构化日志"，但现代可观测性还需**指标**和**追踪**。文档无指标体系：路由各层命中率/降级率、求解耗时分布、LLM 调用次数/成本/token、事件处理延迟、待确认动作积压量。
- **改进**：补"可观测性指标"小节，定义关键 SLI/SLO（路由正确率、求解延迟 P95、事件→处置延迟），并预留 metrics 导出接口（Prometheus 格式）。

### 12. 错误处理与降级策略不系统
- **问题**：设计 13 节只针对 LLM 分类提"解析失败重试一次 + 降级"。嵌入失败、求解器超时/无解/不可行、工具调用失败、事件处理失败、adapter 调用超时的降级与重试策略未系统化。
- **佐证**：`solver_time_limit_s=10.0` 硬编码在策略类，超时行为未定义；事件 handler 失败只 try/except 吞掉。
- **改进**：新增"全链路韧性"章节：每类失败的重试策略（次数/退避）、降级行为（求解无解→诊断+建议人工、嵌入失败→LLM、LLM 失败→澄清）、超时配置化、熔断（adapter 连续失败时）。

### 13. 持久化与状态管理设计薄弱
- **问题**：设计 4.5/4.6 说 Memory/AuditLog"接口可替换持久化"，但未设计持久化方案。会话记忆容量上限/淘汰策略、PendingAction 重启恢复、审计日志留存周期与归档、跨会话长期记忆都未规定。
- **改进**：补"状态与持久化"小节：Memory 的 session 容量与 LRU 淘汰、PendingAction 落库与重启恢复、AuditLog 留存周期与滚动归档；明确 v0.1 内存态的边界与 v0.2 持久化目标。

## P2 — 领域与交互设计缺失

### 14. 领域模型缺状态机定义
- **问题**：`Order`/`WorkOrder` 有 status（draft/dispatched/in_progress/done/blocked），但设计 4.1 未定义合法状态转移图，易写出非法转移（如 done→in_progress）。
- **改进**：补状态机图（Mermaid），明确每个 status 的合法后继与触发动作，并在 WorkOrder 文档约束 dispatch/in_progress 转移的前置条件。

### 15. 排产"可迭代"交互协议不清晰
- **问题**：设计多处提"返回(可迭代)""locked_assignments""excluded_lines"，但未定义迭代协议：用户如何锁定某单的产线/时间、引擎如何增量重排（全量重解 vs 局部修复）、已下发任务令是否冻结、迭代次数上限。
- **改进**：补"排产迭代交互协议"小节，定义锁定/排除的语义、增量重排的求解约束（锁定项作硬约束）、迭代上下文如何在 Memory 传递。

### 16. 求解无解的诊断设计不足
- **问题**：设计 6.5 说"无解时返回明确不可行原因"，但 CP-SAT 求无解时诊断不可行约束本身是难题，文档未给思路。
- **改进**：补求解不可行的诊断策略：CP-SAT 的 `analyze_conflicts` / 冲突约束提取、逐步松弛（先放交期再放产能）定位瓶颈资源、feasibility gap 量化、返回"最接近可行的松弛建议"供人决策。

## P2 — 测试与质量设计缺失

### 17. 测试策略缺契约测试与黄金集
- **问题**：设计 11 节只说"pytest，LLM mock 掉"。但策略插件化的**契约测试模板**（新策略必须满足什么）、MockAdapter→真实 adapter 的**契约测试**（验证"替换零改动"）、路由的**黄金集回归测试**（防示例话术调整引入回归）都未设计。这对验收标准"替换 adapter 零改动""策略插件化"至关重要。
- **改进**：补"测试策略"小节：定义策略契约测试基类、adapter 契约测试套件（同一组用例跑 Mock 和真实）、路由黄金集（标注用例 + 期望路由 + 置信度断言）。

### 18. 验收标准缺量化断言
- **问题**：设计 12 节验收标准多为定性（"判为 query""margin 判据"），示例 E 只说"判为 query 并解释概念"，无置信度/margin 具体断言；策略选择验收也未量化。
- **改进**：把关键验收项量化（如"排产是什么" → query，embedding top1=query 且 margin≥0.10；或 LLM confidence≥0.9）；补路由黄金集的通过率门槛。

## P3 — 部署运维与边界设计缺失

### 19. 缺部署与运维章节
- **问题**：文档无部署方式（单机/容器/进程托管）、配置管理（多环境 .env）、密钥管理、监控告警接入、日志收集、版本升级与数据迁移。
- **改进**：新增"部署与运维"章节，至少覆盖 v0.1 的部署形态与 v0.2 演进方向。

### 20. 缺多租户/并发考虑预留
- **问题**：设计假设单用户，但实际工厂多计划员可能**并发排不同产线**（共享主数据但各自排程），可能产生求解资源竞争与数据一致性问题。v0.2 预留里也未提及。
- **改进**：在 v0.2 预留补"并发排产与会话隔离/锁"考虑，至少设计上不让两个会话同时改同一批任务令。

### 21. 集成层 poll/push 模型衔接未设计
- **问题**：设计 4.2 用 `poll_events()` 拉取模型，但真实 MES/ERP 可能**推送**（webhook/消息队列）。文档未说 adapter 如何同时支持 pull 和 push，以及 poll 间隔与实时性的权衡。
- **改进**：补 adapter 的事件接入模式：pull（巡检间隔配置）+ push（webhook 回调注册），并说明事件去重与顺序保证（当前代码用 `_warned_wo_ids` 去重，可泛化）。

### 22. 权限模型过于粗粒度
- **问题**：设计 4.6 只分 `auto`/`requires_confirmation` 两档 + 读始终允许。实际场景：不同角色权限不同（计划员可 auto 内部催料，实习生需确认；供应商管理员才能确认外部催料）。
- **改进**：在 v0.2 预留补 RBAC：动作 + 资源 + 角色三维授权，v0.1 的两档作为默认策略表的一个维度。

## 与代码实现的关联（落地佐证摘要）

- 设计描述不刚性 → 落地打折的典型案例：第 0 层(#1)、query 锚点(#2) 在代码中完全缺失，根因是文档把"必须项"写成了"建议性描述"。
- 非功能性缺失 → 代码暴露：硬编码密钥(#10)、求解超时硬编码(#12)、全内存无持久化(#13)。
- 职责边界模糊 → 代码耦合：事件层 import 引擎具体类(#7)、validator 双写(#6)、引擎内部关键词分发(#5)。

> 注：以上为**设计文档改进建议**，代码实现层面的问题（如 config.py 密钥、mock 日期过时、澄清启发式粗糙等）已在上轮代码审查报告记录，本文件聚焦设计文档可改进点。
